# Decision Log

This document is the single source of truth for **why** non-trivial implementation
decisions were made in the Blitzy Slack proof-of-concept. Per the project's
Explainability rule, rationale MUST live here — **not** in code comments. A decision
is "non-trivial" if a competent engineer could reasonably have chosen differently.
Any deviation from a literal or obvious interpretation of the requirements MUST have
an explicit entry here; unexplained deviations are treated as defects.

Downstream code-generation agents MUST append a row to the relevant table below for
every additional non-trivial decision they make.

## Root workspace & tooling

| What was decided                                                          | What alternatives existed                                                                                  | Why this choice was made                                                                                                                                                                                                                | What risks it carries                                                                                                                                       |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| pnpm workspace declared with the `packages/*` wildcard in `pnpm-workspace.yaml` | Enumerate each package path explicitly (`packages/db`, `packages/shared`, `packages/api`, `packages/web`) | The wildcard is the AAP/checkpoint-specified workspace contract and automatically includes any future package added under `packages/` without editing the manifest. Explicit enumeration silently omits new packages and drifts from the spec. | A stray non-package directory under `packages/` would be matched by the glob; acceptable because that tree holds only the four intended workspaces. |

## Infrastructure (`docker-compose.yml`)

| What was decided                                                       | What alternatives existed                                              | Why this choice was made                                                                                                                                                                                       | What risks it carries                                                                                                                                                              |
| ---------------------------------------------------------------------- | --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| No top-level `version:` field in `docker-compose.yml`                  | Declare `version: '3.x'` as in legacy Compose files                   | The Compose Specification (Compose v2) ignores `version:` and emits an "obsolete" warning when it is present; omitting it is the current-format default.                                                        | Standalone `docker-compose` v1 binaries would not parse the file, but the project targets Compose v2 (`docker compose`).                                                          |
| `postgres:16-alpine` and `redis:7-alpine` images                       | `postgres:16` / `redis:7` (Debian-based), or pinned patch tags        | Alpine variants are the smallest official images, speeding first-run pulls for the local PoC; the `16`/`7` majors match the AAP-required engine versions.                                                        | Floating on the minor tag means `docker compose pull` can advance the patch level. No native extensions compile against these images, so musl vs glibc is immaterial here.        |
| PostgreSQL initialised as UTF-8 with `C` collation (`POSTGRES_INITDB_ARGS`) | Default locale collation (e.g. `en_US.UTF-8`)                         | `C` collation gives stable byte-level ordering identical across host locales, keeping cursor pagination deterministic between developer machines. Search ranking is handled by `tsvector`, not collation.        | `C` collation sorts bytes, not linguistic order; acceptable because ordering keys are timestamps/ids. Applied only on first volume initialisation.                                |
| Redis started with AOF persistence (`--appendonly yes`)                | RDB snapshotting (default) or no persistence                          | AOF persists presence keys and Socket.io adapter state across container restarts with a tighter durability window than periodic RDB snapshots.                                                                   | AOF files grow larger than RDB and add minor write overhead; negligible at PoC scale.                                                                                            |
| API and web run on the host; only Postgres and Redis run in Compose    | Containerise the API and web servers too                              | Running the Node servers on the host keeps Vite/tsx hot-reload fast and avoids bind-mount/file-watch friction inside containers for the local PoC.                                                               | The host must have Node 22 + pnpm installed (documented in the README). Container/production parity is out of PoC scope.                                                          |
| Named volumes (`blitzy-slack-postgres-data`, `blitzy-slack-redis-data`) | Anonymous volumes, or host bind mounts                                | Stable, predictable names survive `make down`/`make clean` (neither passes `-v`) and are independent of the Compose project directory.                                                                           | Data persists until explicitly removed with `docker compose down -v`; a developer expecting a clean slate must run that command.                                                  |

## `packages/shared` — Shared contracts

| What was decided                                                                                                                                                                                              | What alternatives existed                                                                  | Why this choice was made                                                                                                                                                                                                                                                              | What risks it carries                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| Socket.io event names exported as `as const` string constants (`events.ts`) rather than a TypeScript `enum`                                                                                                   | A TypeScript `enum`, or a plain (non-`as const`) object                                    | `as const` literals erase at compile time (zero runtime cost, no emitted object) and give exact literal types for the typed `ClientToServerEvents`/`ServerToClientEvents` contract; an `enum` emits a runtime object and is awkward to use as a union of literals.                      | Consumers import the constants rather than a runtime enum namespace; this matches how both client and server reference the shared contract.        |
| The error event constant is exported as `ERROR = 'error'` (no `SOCKET_ERROR` alias)                                                                                                                           | Keep the prior `SOCKET_ERROR` name, or export both                                         | The checkpoint/shared contract specifies `ERROR`; socket typings and imports are generated against that name. A second alias would create two ways to reference one wire event and risk drift.                                                                                          | Any code referencing `SOCKET_ERROR` must switch to `ERROR`; no such consumers exist at this checkpoint.                                            |
| `PresenceState` modelled as a string-literal union (`'online' \| 'away' \| 'offline'`)                                                                                                                        | A TypeScript `enum`                                                                        | Same reasoning as the event constants: a union of string literals is zero-runtime, serialises directly over the wire/JSON, and narrows cleanly.                                                                                                                                        | Values are bare strings, so a typo is caught only where the `PresenceState` type is applied; usage sites are typed, so this is contained.          |
| User DTOs modelled as four shapes — `User` (internal, includes `passwordHash`), `UserResponse = Omit<User,'passwordHash'>` (authenticated self), `PublicUser` (peer view, omits `email`/`passwordHash`), `AuthenticatedUser extends PublicUser` (adds `email`) | A single `User` type without `passwordHash`, or overloading `User` for both DB row and wire response | The four-shape contract is the checkpoint/AAP §0.4.4 specification: `passwordHash` stays on the server-only `User` shape, the self-view exposes `email` but no hash, and the peer-view omits `email`. Overloading one type for DB and wire blurs the privacy boundary and risks leaking `passwordHash`. | More named types to keep aligned with the Prisma model; mitigated because all derive from `User` via `Omit`/`extends`, so one source change propagates. |

## `packages/db` — Database package

| What was decided                                                                                                     | What alternatives existed                                                                                      | Why this choice was made                                                                                                                                                                                                                                                                                           | What risks it carries                                                                                                                            |
| -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Singleton `PrismaClient` via a `globalThis` guard in `packages/db/src/client.ts`                                     | (a) A factory function returning fresh instances; (b) A module-level constant only, with no `globalThis` cache | Option (a) fragments the connection pool across test files and breaks Prisma's intended single-instance model; option (b) leaves a window where `tsx watch` and test-runner hot-reload can double-construct the client. The `globalThis` guard handles both production correctness and dev-mode hot-reload safety. | Couples the module to Node-specific `globalThis` semantics. Acceptable because the package is Node-only (server-side).                           |
| Prisma logging fixed to `log: ['warn', 'error']`                                                                     | (a) `['query', 'info', 'warn', 'error']` for verbose dev; (b) `['error']` only                                 | AAP §0.6.1 Group 2 mandates `['warn', 'error']` verbatim. Verbose `query` logging would flood the Pino output and obscure real signals (Gate 10 — Observability). Suppressing warnings would hide deprecations.                                                                                                    | If a hot debugging scenario needs query-level logs, a developer must temporarily edit `client.ts`. Documented here as the trade-off.             |
| Graceful shutdown via an exported `async disconnectPrisma()` plus `process.once('SIGTERM'\|'SIGINT')` handlers that `await` the disconnect (inside a re-entrancy guard) and then re-raise the original signal with `process.kill(process.pid, signal)` | (a) `process.on` with fire-and-forget `void prisma.$disconnect()`; (b) `beforeExit` only; (c) no shutdown hook; (d) export `disconnectPrisma()` with no in-package signal handlers, letting the API layer own all signal handling | `process.on` + `void` (a) swallows disconnect failures and, by registering a listener, overrides Node's default signal termination so the process can hang or surface unhandled rejections; `beforeExit` (b) never fires on signal kill (`docker compose down`, `make down`); (c) orphans PostgreSQL connections, eventually causing "too many connections". `process.once` + `await` guarantees the disconnect runs exactly once and completes, and re-raising the now-removed signal restores default termination. The exported `disconnectPrisma()` additionally lets the API server's top-level shutdown path coordinate HTTP/socket closure with the Prisma disconnect (subsuming option (d)). | A failed disconnect is swallowed best-effort (the logger may already be torn down). Re-raising depends on `once` having removed the handler, so a repeated signal terminates immediately — intended. |
| JSDoc references in `client.ts` use the package name `@app/db` rather than the AAP's illustrative `@blitzy-slack/db` | Keep the literal `@blitzy-slack/db` string from the agent prompt                                               | The actual workspace package name (set during setup in `packages/db/package.json`) is `@app/db`. Documentation must match the real import specifier so consumers are not misled. Functional code is unchanged.                                                                                                     | None functional; this is a documentation-accuracy alignment. If the package is ever renamed, the JSDoc must be updated alongside `package.json`. |

## `packages/db` — Prisma migrations

| What was decided                                                                                                                                                                                              | What alternatives existed                                                                            | Why this choice was made                                                                                                                                                                                                                                                                                                  | What risks it carries                                                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Full-text search column named `contentTsv` (index `Message_contentTsv_idx`)                                                                                                                                   | The earlier `contentSearch` / `Message_contentSearch_idx` names                                      | `contentTsv` is the checkpoint/AAP-specified name that later search queries (`prisma.$queryRaw` in `search.service.ts`) are written against. A mismatched name fails at runtime with a missing-column error.                                                                                                               | The generated column must stay in sync with the Prisma schema authored in CP2; the migration is the source of truth until then.                                                  |
| 1:1 DM uniqueness enforced via sorted `participantOneId`/`participantTwoId` columns on `DirectMessage` plus a unique pair index, retaining `DMParticipant` for membership                                      | (a) Rely only on the `DMParticipant(dmId,userId)` PK; (b) a single canonical `participantKey` string column with a unique index | The `DMParticipant` PK only prevents duplicate participants inside one DM — it does not stop two `DirectMessage` rows holding the same pair, violating the AAP "unique composite on the participant pair". Sorted columns + a unique index give deterministic DB-level enforcement and stay queryable; a concatenated key (b) is opaque and harder to join on. | The service layer MUST canonicalise the pair (sort the two user ids) before insert so the unique index is effective — an application invariant the CP2 service code must honour. |
| Message cursor indexes created `createdAt DESC, id DESC` (the `channelId`/`dmId` composites and the standalone `createdAt` index)                                                                              | Default ascending `createdAt` indexes (relying on PostgreSQL backward scans)                         | Pagination loads newest-first with a `(createdAt, id)` cursor; a matching `DESC, DESC` index represents that contract explicitly and serves the ordering without depending on backward-scan behaviour.                                                                                                                     | Ascending range scans for the rare oldest-first case use the index less optimally; acceptable because the product only paginates newest-first.                                  |
| DB-level `VARCHAR(n)` caps mirroring `packages/shared/src/constants/limits.ts` (`displayName` 80, channel `name` 80 / `description` 250, message `content` 4000, reaction `emoji` 64) plus `VARCHAR(255)` on file `originalName`/`storedName`/`mimeType` | Unbounded `TEXT` everywhere, relying solely on Zod validation at the API                             | DB caps add defense-in-depth against oversized direct writes (DoS/data-integrity) independent of the API. The 80/250/4000/64 values mirror the shared limit constants; 255 is the conventional bound for filesystem names and MIME types. No `CHECK` constraint is added — the forbidden Message channel/DM XOR check is intentionally omitted. | The DB caps must stay in sync with the shared constants if those limits change; the CP2 Prisma schema expresses these via `@db.VarChar(n)`.                                       |
| `File.uploadedById` made nullable with FK `ON DELETE SET NULL ON UPDATE CASCADE`                                                                                                                              | Keep `NOT NULL` with `ON DELETE CASCADE` (delete file metadata when the uploader is deleted)         | The checkpoint requires uploaded-file records to survive deletion of the uploader; `SET NULL` preserves the file row and clears only the author reference, which `CASCADE` cannot do on a non-nullable column.                                                                                                            | A file can exist with no uploader (`uploadedById IS NULL`); consumers and the CP2 Prisma relation must treat the uploader as optional.                                            |

## `packages/api` — Configuration (env loader)

| What was decided                                                                                                       | What alternatives existed                                                        | Why this choice was made                                                                                                                                                                                                                                   | What risks it carries                                                                                                                                                                                                                                         |
| ---------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Zod-validated env loader (`packages/api/src/config/env.ts`) using `safeParse` + `process.exit(1)` on failure           | Throw on validation failure; lazy per-getter validation; no validation at all    | Fail-fast at module load surfaces an immediate, human-readable diagnostic (`error.format()`) before any other module initialises, satisfying Gate 12; `safeParse` lets us format the error and pick the exit code rather than surfacing a raw throw.       | An invalid environment cannot be recovered without a restart — the intended behaviour for a configuration error.                                                                                                                                              |
| `JWT_SECRET` requires `.min(32)` characters (no default)                                                               | No minimum; 16 chars; 64 chars                                                   | 32 bytes is the recommended floor for HS256 signing-key strength (AAP §0.8.4 security baseline).                                                                                                                                                           | Operators must supply a real 32+ char secret; the `.env.example` placeholder is long enough to pass and documents the requirement.                                                                                                                            |
| `BCRYPT_ROUNDS` requires `.min(10)`, default `10`                                                                      | No minimum; default/minimum of 12                                                | 10 is the OWASP-recommended minimum bcrypt cost factor (AAP §0.8.4); keeping it as the default while allowing higher values lets ops tune the work factor via env.                                                                                         | Higher rounds increase login latency; ops can raise the value when hardware allows.                                                                                                                                                                           |
| `CORS_ORIGIN` defaults to `http://localhost:5173`                                                                      | No default (require it); derive it from `VITE_APP_URL`                           | Neither root nor `packages/api/.env.example` declares `CORS_ORIGIN`, so a default equal to the Vite dev origin keeps `make local` working out of the box (Gate 1).                                                                                         | Production deployments must override it; this row is the single source of truth for that obligation.                                                                                                                                                          |
| `console.error` is the only `console` use in the API package, confined to env.ts's fail-fast path                      | A temporary console-based pseudo-logger; a silent exit                           | The Pino logger imports from env.ts, so using it here would be circular and it is not yet constructed; the API ESLint config does not enable `no-console`, so `console.error` is permitted. Every other module uses the Pino logger.                       | One bounded exception, documented here; if `no-console` is later enabled it must allow `error`.                                                                                                                                                               |
| `z.coerce.number()` for the numeric vars `PORT`, `MAX_FILE_SIZE_MB`, `BCRYPT_ROUNDS`                                   | A custom `transform`; `parseInt` + `refine`                                      | `process.env` values are always strings; `z.coerce.number()` is the idiomatic Zod way to convert and then validate via `.int().positive()` / `.min()`.                                                                                                     | None of note.                                                                                                                                                                                                                                                 |
| `.startsWith('postgresql://')` on `DATABASE_URL` and `.startsWith('redis://')` on `REDIS_URL`, in addition to `.url()` | `z.string().url()` alone                                                         | `.url()` accepts any scheme, so a mis-pasted `http://` connection string would pass; the scheme guard rejects that class of typo at startup with a clear message before Prisma/ioredis fail later.                                                         | Belt-and-suspenders validation that overlaps `.url()`; accepted in exchange for an earlier, clearer failure.                                                                                                                                                  |
| Schema field named `PORT` (reads `process.env.PORT`, default `3000`), matching the `env.PORT` export contract          | Read `API_PORT` (the name used in both `.env.example` files); support both names | The file schema and agent prompt specify `PORT`, and consumers (`index.ts` → `httpServer.listen(env.PORT)`) depend on `env.PORT`; the default `3000` converges with the `.env.example` `API_PORT=3000`, so `make local` (Gate 1) works with no `PORT` set. | If an operator sets a non-default `API_PORT` expecting the server to follow, `env.PORT` will not track it; the server port and the Makefile health-check URL should both align on `PORT` (or `PORT` should be set explicitly) when a custom port is required. |
