# Decision Log

This document is the single source of truth for **why** non-trivial implementation
decisions were made in the Blitzy Slack proof-of-concept. Per the project's
Explainability rule, rationale MUST live here — **not** in code comments. A decision
is "non-trivial" if a competent engineer could reasonably have chosen differently.
Any deviation from a literal or obvious interpretation of the requirements MUST have
an explicit entry here; unexplained deviations are treated as defects.

Downstream code-generation agents MUST append a row to the relevant table below for
every additional non-trivial decision they make.

## How to use this log

- **Scope — every non-trivial decision.** A decision is "non-trivial" if a competent engineer could reasonably have chosen differently. Each one gets a row in the relevant section below.
- **Append-only.** Rows are never edited or removed. If a decision is later reversed or refined, add a NEW row in the same section that names the decision it supersedes, rather than rewriting history.
- **Code comments carry HOW, this log carries WHY.** Comments may describe mechanics, edge cases, and non-obvious control flow; rationale and trade-offs belong here. This keeps the "why" out of line-level `git blame` churn and in one auditable place (AAP §0.8.3).
- **Deviations are defects unless logged.** Any departure from a literal or obvious reading of the Agent Action Plan (AAP) MUST have an explicit row explaining it.
- **Organisation.** Decisions are grouped by the package or concern they belong to — one `##` section per area — and each section holds a four-column table: _What was decided_ · _What alternatives existed_ · _Why this choice was made_ · _What risks it carries_. Add a new `##` section for a new area, or append to an existing section when one already fits.
- **Adding a row.** Append to the bottom of the matching section's table, fill all four columns, wrap code identifiers in backticks, and escape any literal pipe inside a cell as `\|`.

## `packages/api` — Configuration (Pino logger, `config/logger.ts`)

| What was decided | What alternatives existed | Why this choice was made | What risks it carries |
| --- | --- | --- | --- |
| Pino as the logging library, exposed as a single `logger` singleton constructed at module load | Winston; Bunyan; bare `console` | Pino is the fastest structured logger, is JSON-first, is widely supported by aggregators (Loki, CloudWatch, Datadog), and has the `pino-http` / `pino-pretty` ecosystem the rest of the API relies on (Gate 10) | Less feature-rich than Winston (no built-in transport fan-out) — acceptable for a PoC; a second sink would need a custom transport |
| `pino-pretty` transport is attached **only** when `NODE_ENV === 'development'`; test and production emit raw JSON to stdout | Pretty output in every environment; never pretty (raw JSON even in dev) | Production logs must stay machine-ingestible JSON, while dev ergonomics need human-readable colorized lines; gating on `NODE_ENV` gives both without a code change | The dev pretty path spawns a `pino-pretty` worker thread on first flush (negligible overhead); the two code paths must be kept behaviourally equivalent apart from formatting |
| Redaction policy redacts `req.headers.authorization`, `req.headers.cookie`, `req.body.password`, `req.body.passwordHash`, `req.body.token`, the bare keys `password` / `passwordHash` / `token`, and their one-level `*.` wildcards, with `censor: '[REDACTED]'` and `remove: false` | Redact nothing (trust producers); redact an exhaustive deny-list of everything that looks sensitive | A conservative, explicit allow-list blocks the known secret-bearing fields (JWTs, cookies, credentials) emitted by `pino-http` and ad-hoc logs while keeping the log shape intact for ingestion (`remove: false`) | A novel secret-bearing field name would slip through; mitigated by this documented policy and by producers not passing secrets explicitly |
| Timestamps use `pino.stdTimeFunctions.isoTime` (ISO 8601 UTC strings) | Pino default Unix-epoch milliseconds; locale-specific formatting | ISO 8601 is sortable, timezone-explicit, and the lingua franca of log aggregators | Marginally larger log lines than an epoch integer (negligible) |
| The level formatter emits the string label (`formatters.level(label) => { level: label }`) so lines carry `"level":"info"` | Pino default numeric levels (`"level":30`) | String labels are human-scannable in a raw `tail` and friendlier to downstream parsing/alerting while preserving structured semantics | Differs from the Pino default; documented here so consumers expect labels, not numbers |
| `env.LOG_LEVEL` (when set) overrides the `NODE_ENV`-derived default (`debug` in development, `info` in test/production), resolved in `resolveLogLevel()` | A `NODE_ENV`-only level with no override hook | Ops can raise or lower verbosity at runtime via env without a code change or redeploy | A forgotten `LOG_LEVEL=trace` (or `silent`) in production would over- or under-log; mitigated by env review and the fail-fast env loader making the value visible |
| `resolveLogLevel()` is annotated `: Level \| 'silent'` instead of the agent prompt's literal `: Level` | Return pino's `Level` verbatim (the prompt's literal); treat `'silent'` as the `NODE_ENV` default and drop it; suppress with a cast | Deviation logged per the Explainability rule: pino 9.x's exported `Level` union omits `'silent'`, but `env.LOG_LEVEL` is validated as `z.enum([...,'silent']).optional()` and may legitimately be `'silent'`. `Level \| 'silent'` is structurally identical to pino's `LevelWithSilent`, is assignable to `LoggerOptions.level` (`LevelWithSilentOrString`), preserves the explicit-`silent` behaviour, and compiles with zero `@ts-ignore` (Rule 3) | If a future pino major adds `'silent'` to `Level`, the union becomes redundant (harmless); the narrowed name `Level` is still imported and used so the schema's import contract holds |
| The static `base.service` log field is the literal `'@blitzy-slack/api'` even though the workspace package is scoped `@app/api` | Use the real package name `@app/api`; derive it from `package.json` `name` at runtime | Deviation logged per the Explainability rule: `service` is a free-form observability label (not a module specifier, so unlike the Jest `moduleNameMapper` keys it has no resolution requirement), the project itself is named `blitzy-slack` (catalog-info.yaml / mkdocs), and the file schema plus the spec's own validation command assert on exactly `"service":"@blitzy-slack/api"` | A reader correlating logs with the npm package name sees `@blitzy-slack/api` rather than `@app/api`; this row is the single source of truth reconciling the two |

## `packages/api` — Request validation middleware (`middleware/validate.ts`)

| What was decided | What alternatives existed | Why this choice was made | What risks it carries |
| --- | --- | --- | --- |
| Validated payloads are funnelled through an explicitly `unknown`-typed local (`const data: unknown = parsed.data; req.body = data;` and `const source: unknown = req[target];`) before being written back to the request | Make `validate` generic over `z.infer<T>`; cast `parsed.data` to a concrete type; relax the `no-unsafe-assignment` lint rule | Zod's `ZodTypeAny.safeParse().data` and Express 5's `req.body`/`req[target]` are both typed `any`, which tripped `@typescript-eslint/no-unsafe-assignment` and failed the module-wide zero-warning gate (Rule 3). Routing the value through `unknown` is the lint-sanctioned way to tame an `any`, keeps the public overload signatures unchanged, and preserves runtime behaviour exactly. Fixed during `logger.ts` validation as the module-level final validator (the file is an in-scope AAP CREATE and had no call sites yet, so the change is zero-blast-radius) | The middleware still does not statically type the parsed output for downstream handlers; a future generic refactor over `z.infer<T>` would carry the schema's type through to `req.body`, and this row should be superseded when that lands |
